## 用户输入

```text
$ARGUMENTS
```

如果用户输入不为空，在继续之前**必须**先理解用户输入的内容。

## 执行流程

用户在 `/specify` 后输入的内容就是功能描述。即使 `$ARGUMENTS` 显示为字面量，也假设它已在本次对话中可用。

### 第一步：生成分支短名

分析功能描述，提取最关键的关键词，生成一个 2-4 个词的分支短名：

- 格式：动词-名词（如 `add-user-auth`、`fix-payment-bug`）
- 保留技术术语和缩写（OAuth2、API、JWT 等）
- 全小写，单词间用连字符
- 不超过 40 个字符
- 示例：
  - "我想添加用户认证功能" → `user-auth`
  - "实现 Memo 标签过滤" → `memo-tag-filter`
  - "修复登录页面的 XSS 漏洞" → `fix-login-xss`

### 第二步：检查是否已有同名分支/目录

```bash
git fetch --all --prune
git branch -a | grep -E '<short-name>'
ls specs/active/ | grep -E '<short-name>'
```

- 如果已存在，继续使用已有的分支和 spec 文件（更新模式）
- 如果不存在，创建新的

### 第三步：确定 SPEC_FILE 路径

```
specs/active/<short-name>.md
```

如果是全新功能，创建分支并初始化文件：

```bash
git checkout -b feat/<short-name>
```

### 第四步：读取模板

读取 `specs/templates/spec-template.md` 了解必填结构，然后读取 `specs/active/` 中现有的 spec 文件了解中文写作风格。

### 第五步：解析并生成 Spec

按以下逻辑处理功能描述：

1. **提取关键要素**
   - **参与者（Actors）**：谁在使用这个功能？
   - **动作（Actions）**：他们要做什么？
   - **数据（Data）**：涉及哪些数据实体和属性？
   - **约束（Constraints）**：有哪些限制或边界条件？

2. **处理模糊内容**
   - 优先做有根据的推断，基于项目上下文和行业惯例填充细节
   - 推断内容用 `「推断」` 标注
   - 仅在以下情况标注 `[待确认：具体问题]`：
     - 选择会显著影响功能范围或用户体验
     - 存在多种合理解读，各有不同含义
     - 没有合理的默认值
   - **最多标注 3 处 `[待确认]`**，超出部分直接推断

3. **填写各章节**
   - 用户场景与测试：2-4 个用户故事，按优先级排列（P1 最关键）
   - 每个用户故事必须**独立可测试**
   - 功能需求：每条必须可测试，描述能力而非实现步骤
   - 成功标准：可度量、技术无关、用户视角（见下方规范）

4. **成功标准写法规范**
   - 必须包含具体数字、比率或时间
   - 不提技术实现细节（不写"接口响应 200ms"，写"用户立即看到结果"）
   - 从用户/业务视角描述结果
   - 好的示例：
     - "用户可在 30 秒内完成一条 Memo 的记录"
     - "标签过滤结果在 1 秒内显示"
     - "90% 的用户能在首次尝试时成功完成主要操作"
   - 不好的示例：
     - "API 响应时间低于 200ms"（技术细节）
     - "数据库查询高效"（不可度量）

### 第六步：将 Spec 写入 SPEC_FILE

使用中文，按模板结构填写，用具体内容替换占位符，保留章节顺序和标题。

### 第七步：质量自校验

生成 Spec 后，逐项检查以下质量标准：

**内容质量**
- [ ] 无实现细节泄露（无编程语言、框架、API 名称）
- [ ] 聚焦用户价值和业务需求
- [ ] 所有必填章节均已完成

**需求完整性**
- [ ] 每条 FR 都有明确的验收条件，可判断通过/失败
- [ ] 成功标准可度量（含具体数字）
- [ ] 成功标准技术无关
- [ ] 所有验收场景已定义
- [ ] 已识别边界场景
- [ ] 功能范围边界清晰
- [ ] 已记录假设和依赖

**如果发现不符合项**：直接修复 Spec，重新检查，最多迭代 3 次。

**如果存在 `[待确认]` 标注**：提取所有标注，按以下格式逐个提问（最多 3 个）：

```markdown
## 问题 [N]：[主题]

**上下文**：[引用相关 spec 段落]

**需要确认的内容**：[具体问题]

**建议选项**：

| 选项 | 内容 | 影响 |
|------|------|------|
| A    | [第一个建议] | [对功能的影响] |
| B    | [第二个建议] | [对功能的影响] |
| 自定义 | 请提供你的答案 | — |

**你的选择**：_[等待回复]_
```

同时提出所有问题，等用户一次性回答后，将回答替换到 spec 中相应的 `[待确认]` 位置，再次运行质量校验。

### 第八步：完成报告

输出：
- 分支名
- Spec 文件路径
- 质量校验结果摘要
- 下一步建议（如触发 SDD pipeline）

## 通用准则

- 聚焦 **是什么**（用户需要什么）和**为什么**（业务价值）
- 避免**怎么做**（不涉及技术栈、API、代码结构）
- Spec 是写给产品/业务视角的人看的，不是写给开发者的
- 参考 `CLAUDE.md` 了解项目的核心价值：低摩擦记录——spec 本身也不应增加不必要的摩擦

## 项目上下文

- **项目**：AIFlomo，Flomo 克隆，支持 Web / Android / iOS
- **技术栈**：Expo (React Native) + Node.js Fastify + SQLite + Drizzle ORM（spec 中不提这些）
- **核心价值**：低摩擦记录，先打通"输入→存储→回看"闭环
- **Spec 保存路径**：`specs/active/<short-name>.md`
- **Spec 语言**：中文
