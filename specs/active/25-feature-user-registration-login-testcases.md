## 用户登录

### 正常场景

- **有效邮箱和密码，登录成功**
  - 操作步骤：
    1. 后台已创建测试账号（邮箱：test@example.com，密码：Password123）
    2. 访问登录页面
    3. 在邮箱输入框输入 `test@example.com`
    4. 在密码输入框输入 `Password123`
    5. 点击"登录"按钮
  - 预期结果：
    - 接口返回 200
    - 响应体：`{ data: { userId, email, nickname }, error: null, message: "登录成功" }`
    - Set-Cookie 响应头包含 Session Cookie，属性为 `httpOnly=true; sameSite=strict`（生产环境还需 `secure=true`）
    - 页面跳转到笔记列表页（/notes 或主页）
    - 数据库 Session 表中新增一条记录，关联到该用户
    - User 表中该用户的 `last_login_at` 字段更新为当前时间

- **邮箱大小写不敏感，登录成功**
  - 操作步骤：
    1. 数据库中用户邮箱为 `test@example.com`（小写）
    2. 访问登录页面
    3. 在邮箱输入框输入 `Test@Example.COM`（混合大小写）
    4. 在密码输入框输入正确密码 `Password123`
    5. 点击"登录"按钮
  - 预期结果：
    - 系统将输入邮箱转为小写后匹配
    - 接口返回 200，登录成功
    - 响应体：`{ data: { userId, email: "test@example.com", nickname }, error: null, message: "登录成功" }`
    - 创建 Session Cookie，跳转到笔记列表页

- **已登录用户再次访问，自动识别登录状态**
  - 操作步骤：
    1. 用户已成功登录，浏览器保存了有效的 Session Cookie
    2. 用户关闭浏览器后重新打开
    3. 访问应用首页（如 /）
  - 预期结果：
    - 系统通过 Cookie 中的 Session Token 识别用户身份
    - 页面直接跳转到笔记列表页，无需重新登录
    - Session 的 `expires_at` 未过期

### 异常场景

- **邮箱不存在，登录失败**
  - 操作步骤：
    1. 访问登录页面
    2. 在邮箱输入框输入 `nonexistent@example.com`（数据库中不存在该用户）
    3. 在密码输入框输入任意密码 `AnyPassword123`
    4. 点击"登录"按钮
  - 预期结果：
    - 接口返回 401
    - 响应体：`{ data: null, error: "邮箱或密码错误", message: "邮箱或密码错误" }`
    - 页面显示提示文字："邮箱或密码错误"
    - 不跳转页面，用户停留在登录页

- **密码错误，登录失败**
  - 操作步骤：
    1. 后台已创建测试账号（邮箱：test@example.com，密码：Password123）
    2. 访问登录页面
    3. 在邮箱输入框输入 `test@example.com`
    4. 在密码输入框输入错误密码 `WrongPassword`
    5. 点击"登录"按钮
  - 预期结果：
    - 接口返回 401
    - 响应体：`{ data: null, error: "邮箱或密码错误", message: "邮箱或密码错误" }`
    - 页面显示提示文字："邮箱或密码错误"
    - 不跳转页面

- **邮箱字段为空，登录失败**
  - 操作步骤：
    1. 访问登录页面
    2. 在邮箱输入框不输入任何内容（留空）
    3. 在密码输入框输入 `Password123`
    4. 点击"登录"按钮
  - 预期结果：
    - 前端验证触发，显示提示："请填写邮箱和密码"
    - 不发送 API 请求
    - 页面不跳转

- **密码字段为空，登录失败**
  - 操作步骤：
    1. 访问登录页面
    2. 在邮箱输入框输入 `test@example.com`
    3. 在密码输入框不输入任何内容（留空）
    4. 点击"登录"按钮
  - 预期结果：
    - 前端验证触发，显示提示："请填写邮箱和密码"
    - 不发送 API 请求
    - 页面不跳转

- **邮箱和密码都为空，登录失败**
  - 操作步骤：
    1. 访问登录页面
    2. 邮箱和密码输入框都不输入任何内容
    3. 点击"登录"按钮
  - 预期结果：
    - 前端验证触发，显示提示："请填写邮箱和密码"
    - 不发送 API 请求

- **邮箱格式不正确，登录失败**
  - 操作步骤：
    1. 访问登录页面
    2. 在邮箱输入框输入格式错误的邮箱 `invalid-email`（缺少 @）
    3. 在密码输入框输入 `Password123`
    4. 点击"登录"按钮
  - 预期结果：
    - 前端验证触发，显示提示："请输入有效邮箱地址"
    - 不发送 API 请求
    - 或如果绕过前端验证，后端返回 400，响应体：`{ data: null, error: "邮箱格式不正确", message: "请输入有效邮箱地址" }`

- **Session 已过期，自动跳转到登录页**
  - 操作步骤：
    1. 用户之前登录过，但 Session 已超过 7 天过期
    2. 访问笔记列表页（需要认证的页面）
  - 预期结果：
    - 系统检测到 Session 过期
    - 页面自动跳转到登录页
    - 显示提示："登录已过期，请重新登录"（可选）

- **未登录访问需要认证的 API，返回 401**
  - 操作步骤：
    1. 浏览器未携带有效 Session Cookie（未登录或已清除 Cookie）
    2. 直接调用需要认证的 API，如 GET /api/notes
  - 预期结果：
    - 接口返回 401
    - 响应体：`{ data: null, error: "未登录", message: "请先登录" }`

## 用户注册

### 正常场景

- **有效信息，勾选协议，注册成功并自动登录**
  - 操作步骤：
    1. 访问注册页面
    2. 在邮箱输入框输入 `newuser@example.com`（数据库中不存在）
    3. 在昵称输入框输入 `小明`
    4. 在密码输入框输入 `Password123`（8-20 位，包含字母和数字）
    5. 勾选"隐私协议"复选框
    6. 点击"注册"按钮
  - 预期结果：
    - 接口返回 201
    - 响应体：`{ data: { userId, email: "newuser@example.com", nickname: "小明" }, error: null, message: "注册成功" }`
    - 数据库 User 表中新增一条记录：
      - `email` = `newuser@example.com`
      - `password_hash` = bcrypt 哈希值（salt rounds = 10）
      - `nickname` = `小明`
      - `created_at` = 当前时间
      - `last_login_at` = 当前时间
    - Set-Cookie 响应头包含 Session Cookie（httpOnly, sameSite strict）
    - 数据库 Session 表中新增一条记录，关联到新用户
    - 页面自动跳转到笔记列表页（无需再次登录）

- **邮箱大小写混合，系统转为小写存储**
  - 操作步骤：
    1. 访问注册页面
    2. 在邮箱输入框输入 `NewUser@Example.COM`（混合大小写）
    3. 在昵称输入框输入 `测试用户`
    4. 在密码输入框输入 `Password123`
    5. 勾选隐私协议，点击"注册"
  - 预期结果：
    - 接口返回 201，注册成功
    - 数据库中存储的邮箱为 `newuser@example.com`（全部小写）

## 安全与边界场景

### 密码存储安全

- **密码使用 bcrypt 哈希存储，不明文保存**
  - 操作步骤：
    1. 用户注册账号，密码为 `Password123`
    2. 查看数据库 User 表中该用户的 `password_hash` 字段
  - 预期结果：
    - `password_hash` 字段值为 bcrypt 哈希值（形如 `$2b$10$...`，长度约 60 字符）
    - 不包含明文密码 `Password123`
    - bcrypt salt rounds = 10

### Session 安全

- **Session Cookie 属性正确配置**
  - 操作步骤：
    1. 用户成功登录
    2. 查看浏览器 Set-Cookie 响应头或开发者工具中的 Cookie
  - 预期结果：
    - Cookie 包含 `httpOnly=true`（JavaScript 无法访问，防止 XSS）
    - Cookie 包含 `sameSite=strict`（防止 CSRF）
    - 生产环境下，Cookie 包含 `secure=true`（仅通过 HTTPS 传输）

- **Session 默认 7 天有效，过期后失效**
  - 操作步骤：
    1. 用户登录后创建 Session
    2. 查看数据库 Session 表中的 `expires_at` 字段
  - 预期结果：
    - `expires_at` = 当前时间 + 7 天
    - 当时间超过 `expires_at` 后，用户访问需要认证的页面，系统拒绝访问并提示重新登录
