# 功能规格：账号注册与登录

**Feature Branch**: `feature/25-user-registration-login`
**创建日期**: 2026-03-04
**状态**: 草稿
**关联 Issue**: #25

## 用户场景与测试

### 用户故事 1 — 邮箱密码登录（优先级：P1）

用户访问应用时，可以通过输入邮箱和密码完成登录，进入主应用界面。这是最核心的身份验证流程，确保已注册用户能够访问其个人笔记数据。

**为什么是 P1**：登录是访问应用的必经之路，是用户回访的核心功能。没有登录，用户无法使用任何笔记功能。

**独立测试方式**：可通过后台直接创建测试账号，然后在登录页面输入凭据验证。不依赖注册流程即可独立测试。

**验收场景**：

1. **假设** 用户已有账号（邮箱：test@example.com，密码：Password123），**当** 用户在登录页输入正确邮箱和密码并点击"登录"，**则** 系统验证成功，创建 Session，跳转到笔记列表页
2. **假设** 用户在登录页面，**当** 用户输入错误的邮箱或密码并点击"登录"，**则** 系统显示"邮箱或密码错误"提示，不跳转页面
3. **假设** 用户在登录页面，**当** 用户未输入邮箱或密码就点击"登录"，**则** 系统显示"请填写邮箱和密码"提示
4. **假设** 用户已登录并保持 Session 有效，**当** 用户再次访问应用，**则** 系统自动识别登录状态，直接进入笔记列表页

---

### 用户故事 2 — 新用户注册（优先级：P2）

新用户可以通过输入邮箱、昵称、密码并勾选隐私协议完成账号注册，注册成功后自动登录进入主应用界面。

**为什么是 P2**：注册是获取新用户的入口，但可以先用后台创建的测试账号验证核心笔记功能（P1 登录已足够支撑）。注册流程完善后，才能真正开放给公众用户。

**独立测试方式**：访问注册页面，输入新邮箱、昵称和密码，勾选协议后提交。验证是否成功创建账号并自动登录。

**验收场景**：

1. **假设** 用户访问注册页面，**当** 用户输入有效邮箱（如 newuser@example.com）、昵称（如"小明"）、密码（8-20位，含字母和数字），勾选隐私协议并点击"注册"，**则** 系统创建账号，自动登录，跳转到笔记列表页
2. **假设** 用户在注册页面，**当** 用户输入已存在的邮箱并点击"注册"，**则** 系统显示"该邮箱已注册，请直接登录"提示
3. **假设** 用户在注册页面，**当** 用户未勾选隐私协议就点击"注册"，**则** 系统显示"请先同意隐私协议"提示，不提交表单
4. **假设** 用户在注册页面，**当** 用户输入的密码少于 8 位或格式不符合要求，**则** 系统显示"密码需为 8-20 位，包含字母和数字"提示
5. **假设** 用户在注册页面，**当** 用户输入的邮箱格式不正确（如缺少 @），**则** 系统显示"请输入有效邮箱地址"提示

---

### 用户故事 3 — 页面间跳转（优先级：P3）

用户可以在登录页和注册页之间自由切换，方便已注册用户快速返回登录，或新用户快速跳转到注册。

**为什么是 P3**：页面跳转是 UX 优化，但不影响核心功能的可用性。用户可以通过手动输入 URL 或浏览器后退达到同样效果，因此优先级较低。

**独立测试方式**：分别在登录页和注册页点击跳转链接，验证页面是否正确切换且不丢失已输入内容「推断：表单清空策略需 UX 确认」。

**验收场景**：

1. **假设** 用户在登录页面，**当** 用户点击"立即注册"，**则** 页面跳转到注册页面
2. **假设** 用户在注册页面，**当** 用户点击"返回登录"，**则** 页面跳转到登录页面

---

### 边界场景

- **邮箱大小写**：系统应将邮箱统一转为小写存储，确保 `User@Example.com` 和 `user@example.com` 视为同一账号
- **密码加密存储**：系统必须使用 bcrypt 或类似哈希算法存储密码，不得明文保存「推断」
- **Session 过期**：Session 默认 7 天有效「推断」，过期后用户需重新登录
- **并发注册**：两个用户同时使用同一邮箱注册时，系统应通过数据库唯一约束确保只有一个成功
- **SQL 注入防护**：所有输入必须通过 Drizzle ORM 参数化查询，不拼接 SQL 字符串
- **XSS 防护**：昵称等用户输入在展示时必须纯文本渲染，不解析 HTML

## 功能需求

### 功能性需求

- **FR-001**：系统必须提供邮箱 + 密码的注册功能，要求邮箱唯一、密码 8-20 位且包含字母和数字
- **FR-002**：系统必须提供邮箱 + 密码的登录功能，验证凭据后创建 Session
- **FR-003**：系统必须要求用户在注册时勾选隐私协议，未勾选不允许提交
- **FR-004**：系统必须支持用户在登录页和注册页之间跳转
- **FR-005**：系统必须在登录成功后创建 Session Cookie（httpOnly: true, sameSite: 'strict', secure: true in production）
- **FR-006**：系统必须在前后端均验证邮箱格式、密码强度、昵称长度（1-20 字符「推断」）
- **FR-007**：系统必须将密码使用 bcrypt 哈希后存储，salt rounds = 10「推断」
- **FR-008**：系统必须在注册成功后自动登录用户，无需再次输入密码
- **FR-009**：系统必须记录用户的创建时间和最后登录时间「推断：用于后续数据分析」
- **FR-010**：系统必须支持 Session 持久化，用户关闭浏览器后再次打开仍保持登录状态（Session 未过期的情况下）

### 核心数据实体

- **用户（User）**：
  - `id`：用户唯一标识（自增主键）
  - `email`：邮箱（唯一索引，小写存储）
  - `password_hash`：密码哈希值（bcrypt）
  - `nickname`：昵称（1-20 字符）
  - `created_at`：注册时间
  - `last_login_at`：最后登录时间

- **会话（Session）**：
  - 使用 Fastify session 插件，Session 数据存储在 SQLite 同库「根据 CLAUDE.md」
  - Session 包含：`user_id`, `expires_at`

## 成功标准

- **SC-001**：用户可在 30 秒内完成注册流程（输入邮箱、昵称、密码、勾选协议、提交）
- **SC-002**：用户可在 15 秒内完成登录流程（输入邮箱、密码、提交）
- **SC-003**：系统在 500ms 内完成登录验证并返回响应「推断：合理的性能目标」
- **SC-004**：注册页面和登录页面 UI 与提供的设计稿一致（邮箱、密码输入框，按钮样式，跳转链接位置）
- **SC-005**：系统能正确拒绝 100% 的无效输入（格式错误的邮箱、弱密码、未勾选协议）
- **SC-006**：系统能正确处理重复邮箱注册，提示"该邮箱已注册"而不是系统错误
- **SC-007**：登录后的 Session 能在 7 天内持续有效，用户无需重复登录
