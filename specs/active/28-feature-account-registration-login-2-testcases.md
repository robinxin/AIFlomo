## 用户注册（POST /api/auth/register）

### 正常场景

- **有效信息+勾选协议，注册成功**
  - 操作步骤：
    1. 访问注册页面 `/register`
    2. 填写邮箱：`user@example.com`，昵称：`张三`，密码：`Pass123!`
    3. 勾选隐私协议复选框
    4. 点击"注册"按钮
  - 预期结果：
    - 接口返回 201
    - 响应体：`{ data: { id, email: "user@example.com", nickname: "张三", createdAt }, message: "注册成功" }`
    - 数据库 `users` 表新增一条记录，`email` 为 `user@example.com`（小写），`passwordHash` 为 bcrypt 哈希值
    - 前端显示"注册成功"提示，1 秒后跳转到登录页面或自动登录并跳转到 `/memo`

- **邮箱包含大写字母，统一转小写存储**
  - 操作步骤：
    1. 访问注册页面
    2. 填写邮箱：`User@Example.com`，昵称：`测试用户`，密码：`Secure123`
    3. 勾选隐私协议
    4. 点击"注册"按钮
  - 预期结果：
    - 接口返回 201
    - 数据库中 `email` 字段存储为 `user@example.com`（全部小写）
    - 后续可以使用 `User@Example.com` 或 `user@example.com` 登录（统一转小写比对）

- **密码为 6 位边界值，注册成功**

## 用户登录（POST /api/auth/login）

### 正常场景

- **有效邮箱和密码，登录成功**
  - 操作步骤：
    1. 假设用户已注册，邮箱：`user@example.com`，密码：`Pass123`
    2. 访问登录页面 `/login`
    3. 填写邮箱：`user@example.com`，密码：`Pass123`
    4. 点击"登录"按钮
  - 预期结果：
    - 接口返回 200
    - 响应体：`{ data: { id, email: "user@example.com", nickname: "张三" }, message: "登录成功" }`
    - 响应头设置 `Set-Cookie: sessionId=xxx; HttpOnly; SameSite=Strict`
    - 数据库 `sessions` 表新增一条记录，`expiresAt` 为当前时间 + 7 天
    - 前端保存用户信息到 `AuthContext`，跳转到 `/memo` 页面

- **邮箱大小写混合，统一转小写比对，登录成功**
  - 操作步骤：
    1. 假设注册时邮箱存储为 `user@example.com`（小写）
    2. 访问登录页面
    3. 填写邮箱：`User@Example.com`（大小写混合），密码：`Pass123`
    4. 点击"登录"按钮
  - 预期结果：
    - 后端将邮箱转为小写 `user@example.com` 后查询
    - 接口返回 200，登录成功

- **Session 有效期为 7 天**
  - 操作步骤：
    1. 用户登录成功
    2. 查询数据库 `sessions` 表中的 `expiresAt` 字段
  - 预期结果：
    - `expiresAt` 的 Unix 时间戳（毫秒）= 当前时间 + 7 天（604,800,000 毫秒）

- **邮箱不存在，返回 401**
  - 操作步骤：
    1. 访问登录页面
    2. 填写邮箱：`nonexistent@example.com`（未注册），密码：`Pass123`
    3. 点击"登录"按钮
  - 预期结果：
    - 接口返回 401
    - 响应体：`{ data: null, error: "INVALID_CREDENTIALS", message: "邮箱或密码错误" }`
    - 不暴露"邮箱不存在"的具体信息（安全要求）

  - 操作步骤：
    1. 假设用户已注册，邮箱：`user@example.com`，正确密码：`Pass123`
    2. 访问登录页面
    3. 填写邮箱：`user@example.com`，密码：`WrongPass`
    4. 点击"登录"按钮
  - 预期结果：
    - 接口返回 401
    - 响应体：`{ data: null, error: "INVALID_CREDENTIALS", message: "邮箱或密码错误" }`

## 用户登出（POST /api/auth/logout）

### 正常场景

- **已登录用户登出成功**
  - 操作步骤：
    1. 用户已登录，Session 有效
    2. 发送 POST /api/auth/logout 请求（携带有效 Session Cookie）
  - 预期结果：
    - 接口返回 204（无响应体）
    - 数据库中对应的 Session 记录被删除
    - 响应头包含 `Set-Cookie: sessionId=; Max-Age=0`（清除 Cookie）
    - 前端清除 `AuthContext` 中的 `user` 数据

### 异常场景

- **未登录时访问登出接口，返回 401**
  - 操作步骤：
    1. 不携带 Session Cookie
    2. 发送 POST /api/auth/logout 请求
  - 预期结果：
    - 接口返回 401
    - 响应体：`{ data: null, error: "Unauthorized", message: "请先登录" }`

- **Session 已过期时访问登出接口，返回 401**
  - 操作步骤：
    1. 假设 Session 已过期（`expiresAt` < 当前时间）
    2. 携带过期的 Session Cookie，发送 POST /api/auth/logout 请求
  - 预期结果：
    - 接口返回 401
    - 响应体：`{ data: null, error: "Unauthorized", message: "请先登录" }`

## 前端页面导航

  - 操作步骤：

    - 页面跳转到 `/register`

- **从注册页面点击"返回登录"，跳转到登录页面**
  - 操作步骤：
    1. 访问注册页面 `/register`
    2. 点击"返回登录"链接
  - 预期结果：
    - 页面跳转到 `/login`
    - 登录表单为空白状态

- **注册成功后跳转到登录页面或自动登录**
  - 操作步骤：
    1. 在注册页面填写有效信息并提交
    2. 注册成功
  - 预期结果：
    - 显示"注册成功"提示
    - 1 秒后自动跳转到 `/login` 或直接跳转到 `/memo`（如果自动登录）

- **登录成功后跳转到主页面**
  - 操作步骤：
    1. 在登录页面填写有效信息并提交
    2. 登录成功
  - 预期结果：
    - 页面跳转到 `/memo`（主应用页面）

### 异常场景

- **在登录/注册页面之间来回切换，之前填写的表单数据不保留**

    1. 访问登录页面，填写邮箱：`test@example.com`

## Session 安全

### 正常场景

- **Cookie 设置为 HttpOnly**
  - 操作步骤：
    1. 用户登录成功
    2. 检查响应头中的 `Set-Cookie` 字段
  - 预期结果：
    - Cookie 包含 `HttpOnly` 标志
    - 前端 JavaScript 无法通过 `document.cookie` 读取 Session ID

- **Cookie 设置为 SameSite=Strict**
  - 操作步骤：
    1. 用户登录成功
    2. 检查响应头中的 `Set-Cookie` 字段
  - 预期结果：
    - Cookie 包含 `SameSite=Strict` 标志
    - 防止 CSRF 攻击

- **生产环境 Cookie 设置为 Secure**
  - 操作步骤：
    1. 设置 `NODE_ENV=production`
    2. 用户登录成功
    3. 检查响应头中的 `Set-Cookie` 字段
  - 预期结果：
    - Cookie 包含 `Secure` 标志
    - 仅通过 HTTPS 传输

- **密码使用 bcrypt 存储，salt rounds = 10**
  - 操作步骤：
    1. 用户注册成功
    2. 查询数据库 `users` 表中的 `password_hash` 字段
  - 预期结果：
    - 字段值以 `$2b$10$` 开头（bcrypt 哈希格式，salt rounds = 10）
    - 不存储明文密码
